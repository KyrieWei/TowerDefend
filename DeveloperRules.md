# 资深程序开发工程师规范要求

## 1. 代码质量洁癖要求

### 1.1 代码质量
- **函数名**：使用最恰当的命名方式，方便阅读
- **类名**：使用最恰当的命名方式，方便阅读
- **变量名**：使用最恰当的命名方式，方便阅读

### 1.2 代码风格规范
- **命名规范**：采用PascalCase（类名、函数名、局部变量名），遵循UE命名约定
- **语义化命名**：函数参数命名必须反映其业务角色，而非类型名缩写（例如：上下文参数应体现其所处的管线阶段，而非简单缩写类型名）
- **缩进**：统一使用4个空格，禁止使用Tab
- **大括号**：采用Allman风格（大括号独占一行）
- **缩进匹配**：在大括号内部统一增加缩进，在没有大括号的情况下代码缩进保持一致

### 1.3 代码整洁度
- **单一职责**：每个函数/类只做一件事
- **函数长度**：单个函数不超过50行
- **类长度**：单个类不超过500行
- **圈复杂度**：函数圈复杂度不超过10

### 1.4 错误处理
- **异常处理**：必须处理所有可能的异常情况
- **资源管理**：使用RAII模式管理非UObject资源，UObject资源遵循UE GC生命周期
- **空指针检查**：所有指针使用前必须进行有效性检查，使用 check/ensure 宏体系
- **生命周期检查**：所有指针使用前必须确保生命周期有效，不会出现野指针

### 1.5 文件格式
- **文件格式**：文件统一都用UTF-8的格式

### 1.6 注释要求
- **注释缩进**：需要和下面的代码统一一样的缩进
- **类的注释**：每个类都需要有整体功能的概况注释，注释放到头文件里面，不要放到cpp里面
- **函数头注释**：函数如果较复杂或者函数名不能直接表达函数含义，必须在函数上加注释，需要写在头文件内，不要放到cpp里面
- **函数体注释**：复杂逻辑必须有行内注释
- **关键属性注释**：带UPROPERTY的属性前必须有使用说明类注释

## 2. 性能优化要求

### 2.1 内存管理
- **内存泄漏**：零容忍，必须使用内存检测工具定期检查
- **内存分配**：避免频繁的小对象分配，使用对象池
- **缓存友好**：优化数据布局，提高缓存命中率

### 2.2 算法复杂度
- **时间复杂度**：核心算法必须达到最优或次优复杂度
- **空间复杂度**：在保证性能的前提下尽量降低空间复杂度
- **大数据处理**：必须考虑大数据量下的性能表现

### 2.3 并发性能
- **线程安全**：多线程环境下的数据访问必须保证线程安全
- **锁粒度**：尽量使用细粒度锁，避免锁竞争
- **异步处理**：IO密集型操作必须使用异步处理

## 3. 架构设计原则

### 3.1 设计模式应用
- **SOLID原则**：严格遵守单一职责、开闭原则等
- **依赖注入**：使用依赖注入降低耦合度
- **接口隔离**：接口设计要小而专一

### 3.2 模块化设计
- **高内聚低耦合**：模块内部高度内聚，模块间低耦合
- **接口设计**：提供稳定、清晰的接口
- **扩展性**：设计时要考虑未来的扩展需求

### 3.3 逻辑设计
- **数据范围**：所有变量都需要有明确的数据范围，保证不会出错
- **覆盖边缘样例**：用户的输入多样，需要把所有边缘样例都考虑到
- **数据检查**：很多数据的输入，基于使用情况进行多级别检查：区分发布时的检查、编译时的检查、运行时的检查，检查需补充完整的log输出
- **批量修改安全**：任何批量替换操作必须采用隔离保护策略（如 Placeholder 技术），防止误改类型名、注释等不应变动的部分

## 4. 测试要求

### 4.1 编译与验证
- **编译验证**：所有代码修改必须在目标平台编译通过，零错误零警告
- **逻辑验证**：无法直接运行的模块，必须通过代码走读+模拟输入分析逻辑正确性
- **增量验证**：基于编译器反馈迭代修复，直到零错误零警告
- **边界测试**：必须包含边界条件测试
- **异常测试**：必须包含异常情况测试

### 4.2 集成测试
- **端到端测试**：关键业务流程必须有端到端测试
- **性能测试**：必须包含性能基准测试
- **压力测试**：必须包含压力测试场景

### 4.3 模拟测试
- **测试规则**：无法直接执行的模块，必须在实现之后模拟输入并分析执行流程，逻辑上判断正确才能交付

### 4.4 测试总结
- **错误修正**：如果测试不通过，需进行总结分析错误原因，并进行修正
- **修正迭代**：因测试错误引发的修正过程，必须重新回顾需求并修复，且修复过程后必须再次测试

## 5. 安全要求

### 5.1 UE 内存安全
- **弱引用**：优先使用 TWeakObjectPtr / TSoftObjectPtr 避免悬挂引用
- **GC 安全**：所有 UObject 指针必须用 UPROPERTY 标记或手动 AddToRoot，防止被GC回收
- **生命周期**：跨帧持有的 Actor 引用必须使用 IsValid() 检查
- **资源管理**：使用RAII模式管理非UObject资源，UObject资源遵循UE GC生命周期

### 5.2 输入验证
- **数据验证**：所有外部输入必须进行严格验证
- **指针检查**：所有指针使用前必须进行有效性检查，使用 check/ensure 宏体系
- **数据合法性**：数值参数必须进行范围检查（如FOV、BlendWeight等），防止非法值传播

## 6. 性能监控要求

### 6.1 UE 性能分析
- **帧率监控**：使用 SCOPE_CYCLE_COUNTER / SCOPED_NAMED_EVENT 标记关键路径
- **内存监控**：使用 Unreal Insights / LLM 追踪内存分配
- **热路径性能**：关键求值链路耗时不超过 0.5ms，使用 FMemMark 优化临时内存分配

### 6.2 告警机制
- **阈值设置**：设置合理的监控阈值
- **数据校验**：求值结果必须包含有效性检查，无效时输出 Error 日志并回退到安全默认值
- **应急响应**：制定应急响应流程

## 7. 持续学习要求

### 7.1 技术更新
- **技术调研**：定期进行新技术调研
- **技术分享**：定期组织技术分享会
- **技术实践**：在合适场景下尝试新技术

### 7.2 代码评审学习
- **互相学习**：通过代码评审互相学习
- **最佳实践**：总结和推广最佳实践
- **经验分享**：定期分享开发经验

## 8. 工程方法论（AI 协作强制执行）

> 以下方法论从实际工程实践中提炼而来，是 AI 助手在进行代码修改时**必须遵循**的可执行检查清单。

### 8.1 「先查后改」原则

**核心**：任何批量修改前，必须先搜索所有匹配项，按语义分类，明确"改"和"不改"的边界，然后再选择修改策略。

**执行流程**：
1. 用 grep / codebase_search 搜索所有包含目标字符串的文件和行
2. 将搜索结果按语义分类（类型名引用 vs 变量引用 vs 注释引用）
3. 对每一类确定不同的处理策略（改/不改/特殊处理）
4. 选择安全的替换策略（如 Placeholder 隔离保护）

**防止的错误**：类型名被误改、注释被误改、跨语义边界的替换副作用。

### 8.2 「签名即真理」原则

**核心**：修复函数体内的变量引用时，必须先确认该函数的签名中参数叫什么名字，以签名为唯一真理源，不能凭记忆或惯性推断。

**三步修复法**：
1. **读签名**：先确认函数的完整签名（参数类型和参数名）
2. **搜引用**：搜索函数体内所有对旧变量名的引用
3. **逐个替换**：对比签名中的参数名，确认替换目标后再执行

**三级约束链**：
```
类型名（如 FXxxContext） → 参数名（如 XxxContext） → 函数体引用（XxxContext）
```

**防止的错误**：参数名混淆、跨函数的命名惯性导致误替换。

### 8.3 「全量扫描」原则

**核心**：当发现一个模式性的错误时，立即在整个相关目录下搜索同类问题，一次性修复所有实例，而不是等编译器逐个报出来。

**执行流程**：
1. 识别到第一个错误时，判断是否为**系统性模式**（同类型的文件/函数是否都存在同样问题）
2. 如果是系统性问题，立即全量搜索整个代码库
3. 建立完整的修改清单，一次性修复所有实例
4. 修复后再次全量搜索，确认无遗漏

**防止的错误**：遗漏同类文件、多轮往返浪费时间（被动等编译器报错 vs 主动出击）。

### 8.4 「分类替换」原则

**核心**：不同的替换目标必须分别处理，禁止用一刀切的全局替换。

**执行策略**：
1. 按函数签名中的参数类型对所有替换点分类
2. 每一类使用独立的替换规则
3. 对于可能误伤的情况，使用 Placeholder 技术隔离保护

**Placeholder 技术示例**：
```
Step 1: FVisionerViewContext → __PLACEHOLDER_TYPE__（保护类型名）
Step 2: ViewContext → EvaluateContext（替换变量名）
Step 3: __PLACEHOLDER_TYPE__ → FVisionerViewContext（恢复类型名）
```

**防止的错误**：跨函数误替换、类型名与变量名混淆。

### 8.5 「替换后验证」原则

**核心**：任何批量替换完成后，必须通过搜索验证无误改和无遗漏。

**验证检查清单**：
1. [表情] 搜索确认目标替换已全部完成（无遗漏）
2. [表情] 搜索确认类型名/注释等不应修改的内容未被误改
3. [表情] 搜索确认无新增的编译错误模式
4. [表情] 逐文件检查函数体内的变量名与函数签名是否一致

### 8.6 「类型即契约」方法论

**核心**：在使用类型分层的框架中，上下文类型本身就是阶段的契约。变量命名必须与类型的契约语义对齐。

**应用场景**：当框架使用不同类型区分不同管线阶段时（如 InitialContext / UpdateContext / EvaluateContext），每种类型携带的数据不同，变量名即是阶段的人类可读标签。在编译器层面就能防止跨阶段的数据误用。

**执行要求**：
- 函数参数名必须反映所处的管线阶段
- 不同阶段的函数严禁使用相同的泛称参数名
- 基类类型的参数可以使用通用名称（如 `Context`），派生类型必须使用具体阶段名

### 8.7 「错误驱动的认知收敛」方法论

**核心**：通过编译错误修复过程逆向推导框架的设计空间，将零散的局部理解收敛为系统性的架构认知。

**认知演进三阶段**：

| 阶段 | 特征 | 行为 |
|------|------|------|
| **阶段一：机械修复** | 只知道"A改成B" | 被动响应编译错误 |
| **阶段二：模式识别** | 识别到跨文件的规律 | 能按函数签名推断修改目标 |
| **阶段三：架构洞察** | 理解设计决策背后的"为什么" | 能预测新模块的结构 |

**自检要求**：
- 每次修复后问自己：我处于哪个阶段？
- 如果还在阶段一，必须主动搜索更多代码来理解全貌
- 目标是每次任务结束时至少达到阶段二

### 8.8 「预测-验证」自驱学习循环

**核心**：通过"对框架行为做预测 → 与实际代码验证"的循环，不依赖外部反馈也能持续提升对框架的理解。

**执行流程**：
```
遇到新模块 → 基于已知模型做预测（它应该有哪些接口/数据流/阶段？）
    → 阅读实际代码验证
    → 匹配 → 扩大信任区域
    → 不匹配 → 定位偏差（哪个假设被证伪？）→ 修正模型
    → 检查是否揭示了更深的设计模式
    → 更新心智模型 → 回到起点
```

**关键信念**：不匹配比匹配更有价值——错误是学习的燃料，不是代价。

### 8.9 「动手前三问」强制检查

**核心**：在每次代码修改前，强制回答三个问题，防止"动手太快、分析太浅"。

**三个必答问题**：
1. 这个问题的**完整范围**是什么？（不只是当前文件，是所有相关文件）
2. 每个修改点的**正确目标值**是什么？（不是凭记忆，是看签名/看定义）
3. 我的修改策略**有没有副作用**？（类型名/注释/其他模块会不会被误改？）

**只有三个问题都能清晰回答后，才能开始修改。**

---

**总结**：资深开发工程师不仅要写出高质量的代码，还要具备良好的工程素养、团队协作能力和持续学习能力。以上规范是成为优秀开发工程师的基本要求，需要在实际工作中不断实践和完善。第8章的工程方法论是从实际编码协作中提炼出的可执行原则，AI 助手在进行任何代码修改时必须严格遵循。
